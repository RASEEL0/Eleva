# Eleva
AI-powered career copilot — upload a resume, name a target job, and get a match score, a personalized learning plan, resume rewrite suggestions, and interview prep, generated by a pipeline of cooperating AI agents.
Built as a capstone project for an Advanced Agentic AI Systems Engineering course.

# What it does
	1.	You upload a resume (PDF) and a target job title.
	2.	Eleva's agent pipeline reads the resume, researches what the role typically requires, and scores the match.
	3.	Depending on the score, you either get:
	•	80% or higher → an application-ready package (talking points, checklist), or
	•	below 80% → a personalized learning plan closing the gap (skills, tasks, time estimates, project ideas).
	4.	Every candidate — regardless of score — also gets rewritten resume bullet points and a set of interview questions (technical, project-specific, behavioral) with expected answers.
	5.	If you're logged in, every analysis is saved so you can revisit it later from your dashboard.

## Architecture
A single FastAPI backend orchestrates a LangGraph multi-agent pipeline, backed by a locally-running LLM. A lightweight static HTML/CSS/JS frontend talks to it over a plain REST API.

```text
       resume upload
            │
            ▼
┌───────────────────┐
│  Resume Analysis  │ extracts skills, education, experience, strengths/weaknesses
└───────────────────┘
            │
            ▼
┌───────────────────┐
│  Job Requirements │ generates the target role's expected skill profile
└───────────────────┘
            │
            ▼
┌───────────────────┐
│     Skill Gap     │ compares resume vs. requirements → match score
└───────────────────┘
            │
      ┌─────┴─────┐
      ▼           ▼
   ≥ 80%       < 80%
      │           │
┌───────────┐ ┌───────────────────┐ 
│Application│ │  Learning Planner │ Application Ready: confirms qualification and readiness for the role (match score ≥ 80%)
│   Ready   │ │ (skills, tasks,   │ 
└───────────┘ │  time estimates)  │ Learning Planner: generates targeted skills, tasks, and time estimates to bridge the skill gap
      │       └───────────────────┘
      └─────┬─────┘
            ▼
┌───────────────────────┐
│  Resume Improvement   │ rewritten summary, bullet points, keywords
└───────────────────────┘
            │
            ▼
┌───────────────────────┐
│    Interview Prep     │ technical / project / behavioral Q&A
└───────────────────────┘
            ▼ 
END Both branches (application-ready and learning-plan) converge on the same resume-improvement → interview-prep tail, so every candidate gets a complete package regardless of their current match score.
We containerized the API with Docker Compose for reproducible deployment



 # Tech Stack

- fastAPI — backend API framework; chosen for async support and automatic
  request validation, which suits a multi-step AI pipeline well.

- LangGraph — orchestrates the multi-agent pipeline as a state graph
  (resume → job analysis → skill gap → learning/application → interview),
  with conditional routing based on match score.

- Ollama + qwen2.5:7b — runs the LLM locally instead of a paid API,
  avoiding per-token costs and keeping candidate resume data on our own
  infrastructure rather than sending it to a third party.

- SQLAlchemy + SQLite — stores user accounts and analysis history;
  SQLite chosen for zero external setup, appropriate for a capstone-scale
  deployment.

- Passlib (bcrypt) — hashes passwords before storage; industry-standard,
  avoids ever storing plaintext credentials.

- pypdf — extracts raw text from uploaded resume PDFs before sending
  it to the LLM for structured analysis.

- Docker / Docker Compose** — containerizes the app for reproducible,
  one-command deployment (`docker compose up`) instead of manual
  environment setup.

- ngrok — exposes the locally-running backend to the internet with a
  public HTTPS URL, without needing a hosted server for the demo.

Agent	Role	Output
Resume Analysis	Parses the PDF and extracts every skill, education entry, work experience, and project, plus an ATS score and strengths/weaknesses	resume_data, ats_score, strengths, weaknesses
Job Requirements	Given the target job title, generates a realistic required/preferred skill profile and typical responsibilities	required_skills
Skill Gap	Pure Python (no LLM call) — token-based comparison between resume skills and required skills	match_score, missing_skills, routing decision
Application Ready	For candidates at or above the match threshold — application summary, talking points, checklist	application_package
Learning Planner	For candidates below the threshold — a skill-by-skill roadmap with reasoning, tasks, and time estimates, plus project ideas	learning_plan
Resume Improvement	Rewrites the professional summary and weak bullet points, suggests keywords to add	resume_improvements
Interview Prep	Generates technical, project-specific, and behavioral interview questions with expected answers	interview_prep

# Setup
1. Install Ollama and pull the model:
ollama pull qwen2.5:7b
2. Install Python dependencies:
pip install fastapi uvicorn python-multipart pypdf langchain-ollama langgraph sqlalchemy passlib "bcrypt==4.0.1" --break-system-packages
3. Run the backend:
uvicorn main:app --reload
4. Serve the frontend: Open index.html directly in a browser, or serve it as a static file. Set BACKEND_URL near the bottom of the script to wherever the backend is reachable (a local URL, an ngrok tunnel, or a deployed cloud URL).

# API reference
Method	Endpoint	Auth required	Description
POST	/career/analyze	optional	Upload a resume + target job, run the full pipeline. Saves to history if a valid session token is sent.
GET	/career/history	required	List the logged-in user's past analyses (id, job title, score, date).
GET	/career/history/{id}	required	Full stored result for one past analysis.
POST	/register	—	Create an account (email, password as form fields).
POST	/login	—	Returns a session token on success.
POST	/logout	—	Invalidates the given token.
GET	/me	required	Returns the logged-in user's email.
Known limitations & roadmap
	•	Sessions are in-memory — restarting the server logs everyone out. Moving sessions into a DB table (or switching to JWT) removes this.
	•	Skill matching is keyword/token-based, not semantic — it handles phrasing differences reasonably well but isn't a full language-level comparison. An LLM-based matching step is a possible upgrade.
	•	Docker Compose — containerizing the backend, Ollama, and the database into one docker-compose up is the next infrastructure step, ahead of cloud deployment.
	•	Cloud deployment — planned once Docker Compose is in place, on a basic VM (or a managed platform for the backend, with Ollama on a separate always-on host given its memory requirements).
